# Applicant-Level Integration Plan

## Objective

The purpose of this step is to convert raw relational Home Credit tables into clean applicant-level feature blocks.

Final dataset grain:

| Final rule                           | Meaning                                                            |
| ------------------------------------ | ------------------------------------------------------------------ |
| `1 SK_ID_CURR = 1 applicant = 1 row` | The final train and test datasets must keep one row per applicant. |

This step is not full feature engineering.
The goal is to create safe, compact, integration-ready feature blocks from historical tables.

---

## Core Principle

| Principle         | Decision                                                                  |
| ----------------- | ------------------------------------------------------------------------- |
| Base table        | `application_train` and `application_test` are already applicant-level.   |
| Final key         | Every external feature block must end at `SK_ID_CURR` level.              |
| Historical tables | Raw one-to-many tables must not be directly merged with application data. |
| Target usage      | `TARGET` must not be used during aggregation.                             |
| Merge type        | Use left merge from application base to historical feature blocks.        |
| Row safety        | Row count must not change after merging any block.                        |

---

## Rechecked Mistakes to Avoid

| Possible mistake                                                                                                      | Correct handling                                                                                                              |
| --------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| Directly merging raw `bureau`, `previous_application`, POS, installments, or credit-card tables with application data | Aggregate each table first to `SK_ID_CURR` level.                                                                             |
| Treating repeated `SK_ID_CURR` in historical tables as bad duplicate                                                  | Repetition is expected because historical tables store multiple records per applicant.                                        |
| Connecting all POS/installment/credit-card tables only through `previous_application`                                 | These tables already contain `SK_ID_CURR`; `SK_ID_PREV` is useful for loan-level summaries, but not mandatory for connection. |
| Merging `bureau_balance` directly with application                                                                    | Impossible directly because it only has `SK_ID_BUREAU`; it must connect through `bureau`.                                     |
| Filling all missing values immediately after merge                                                                    | Do not do this here. Missing block values may simply mean no historical record.                                               |
| Creating too many features from every category                                                                        | Keep compact and useful summaries only.                                                                                       |
| Treating `sample_submission` as a modeling table                                                                      | It is only for final submission format, not training/integration.                                                             |

## Table Relationship Map

| Table                   | Key columns                  | Current grain                        | Integration route                                                             | Final key    |
| ----------------------- | ---------------------------- | ------------------------------------ | ----------------------------------------------------------------------------- | ------------ |
| `application_train`     | `SK_ID_CURR`, `TARGET`       | One current applicant                | Base train table                                                              | `SK_ID_CURR` |
| `application_test`      | `SK_ID_CURR`                 | One current applicant                | Base test table                                                               | `SK_ID_CURR` |
| `bureau`                | `SK_ID_CURR`, `SK_ID_BUREAU` | One external credit record           | Aggregate by applicant                                                        | `SK_ID_CURR` |
| `bureau_balance`        | `SK_ID_BUREAU`               | Monthly bureau-status history        | Aggregate by `SK_ID_BUREAU`, merge into `bureau`, then aggregate by applicant | `SK_ID_CURR` |
| `previous_application`  | `SK_ID_CURR`, `SK_ID_PREV`   | One previous Home Credit application | Aggregate by applicant                                                        | `SK_ID_CURR` |
| `POS_CASH_balance`      | `SK_ID_CURR`, `SK_ID_PREV`   | Monthly POS/cash-loan history        | Aggregate monthly behavior by applicant                                       | `SK_ID_CURR` |
| `installments_payments` | `SK_ID_CURR`, `SK_ID_PREV`   | Installment/payment records          | Derive simple repayment info, then aggregate by applicant                     | `SK_ID_CURR` |
| `credit_card_balance`   | `SK_ID_CURR`, `SK_ID_PREV`   | Monthly credit-card history          | Aggregate monthly card behavior by applicant                                  | `SK_ID_CURR` |

| Block name             | Source table(s)                         | Purpose                                | Prefix           |
| ---------------------- | --------------------------------------- | -------------------------------------- | ---------------- |

## 1. Application Base

| Item           | Plan                                    |
| -------------- | --------------------------------------- |
| Source         | `application_train`, `application_test` |
| Key            | `SK_ID_CURR`                            |
| Aggregation    | Not needed                              |
| Role           | Main train/test base                    |
| Important note | `TARGET` stays only in train.           |

Application files are already at the correct applicant level.
All historical feature blocks will be merged into these tables using `SK_ID_CURR`.

## Final Merge Order

| Order | Feature block          |
| ----- | ---------------------- |
| 1     | `app_base`             |
| 2     | `bureau_features`      |
| 3     | `previous_features`    |
| 4     | `pos_features`         |
| 5     | `installment_features` |
| 6     | `credit_card_features` |

Merge rule:

| Rule                 | Decision                                              |
| -------------------- | ----------------------------------------------------- |
| Merge type           | Left merge                                            |
| Merge key            | `SK_ID_CURR`                                          |
| Row count            | Must remain equal to application base row count       |
| ID uniqueness        | `SK_ID_CURR` must remain unique after every merge     |
| Missing block values | Keep for now; handle later during cleaning/imputation |

---

## Integration Quality Checks

After creating and merging each block, record:

| Check                             | Purpose                                     |
| --------------------------------- | ------------------------------------------- |
| block row count                   | Check how many applicants have that history |
| block feature count               | Track how many features were added          |
| base row count before/after merge | Detect row explosion or row loss            |
| `SK_ID_CURR` uniqueness           | Preserve applicant-level grain              |
| train/test column alignment       | Keep modeling data consistent               |

## Final Decision

The integration will be done using a compact applicant-level design.
Application tables remain the base, and each historical table becomes a small `SK_ID_CURR`-level feature block before merging.

This prevents row explosion, avoids unnecessary notebook complexity, and keeps the final modeling grain unchanged:

```text
1 SK_ID_CURR = 1 applicant = 1 row
```
